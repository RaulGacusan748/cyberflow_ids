1.2 Data Science Objective & Evaluation Paradigm
We frame this operational challenge as a Supervised Cost-Sensitive Binary Classification task. For any incoming network flow record $x_i$, our classification model $f(x_i)$ must output a binary state prediction:
$$y_i \in \{0, 1\}$$
Where:
$y_i = 0$ represents Benign background connection traffic.
$y_i = 1$ represents an active Intrusion/Attack Vector (e.g., DDoS, Scan).
In network telemetry, benign data overwhelmingly dominates the communication lines, with malicious attacks representing less than $17\%$ of overall transactions. Standard classifiers trained on such imbalanced datasets tend to default to the majority class to minimize raw error rates. Consequently, our primary performance metric is the $F_1$-Score, which represents the harmonic mean of Precision and Recall:
$$F_1 = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}$$
To protect the Security Operations Center (SOC) from alert fatigue, we also actively optimize for Precision:
$$\text{Precision} = \frac{TP}{TP + FP}$$
A high-precision engine guarantees that when an alert is fired, the probability of it being a false alarm is minimized, ensuring that security analysts only spend active cycles triaging real, high-severity threats.
2. Preprocessing, EDA, & Engineering
2.1 Data Strategy & Ingestion Layer
The raw telemetry is derived from the industry-standard University of New Brunswick (UNB) CIC-IDS2017 dataset, which captures realistic background traffic interspersed with common volumetric and protocol-based attacks.
Due to the immense volume of raw packet data, running in-memory operations on the complete dataset can exhaust typical hardware systems. To solve this, our ingestion pipeline employs a memory-safe chunk-processing design (handling $100,000$ rows at a time) combined with a $5\%$ Stratified Downsampling Strategy. This yields an optimized validation dataset of $126,038$ flows while perfectly preserving the minority attack distribution ($104,764$ benign flows, or $83.1\%$, and $21,274$ attack flows, or $16.9\%$).
2.2 Feature Engineering: The Packet Asymmetry Metric
Volumetric denial-of-service floods and rapid port sweeps are characterized by highly asymmetrical communication profiles. An attacker sprays packets at a target, but because they are spoofing IPs or ignoring the transport-layer handshakes, very few (or zero) backward reply packets are returned.
To isolate this behavioral footprint, we engineered a custom, dimensionless Packet Asymmetry Metric:
$$\text{Packet Asymmetry} = \frac{\text{Packets}_{\text{fwd}} - \text{Packets}_{\text{bwd}}}{\text{Packets}_{\text{fwd}} + \text{Packets}_{\text{bwd}} + \epsilon}$$
Where:
$\text{Packets}_{\text{fwd}}$ represents the total count of packets sent in the forward direction.
$\text{Packets}_{\text{bwd}}$ represents the total count of packets sent in the backward direction.
$\epsilon = 10^{-5}$ represents a numeric smoothing parameter that acts as a guardrail against division-by-zero errors when connection-less sweeps generate zero backward exchanges.
2.3 The Leakage Shield Engine
In machine learning, target proxy leakage occurs when a feature in the training dataset contains information that would not actually be available at the time of prediction in a live production environment. For instance, in network logging, certain post-incident flags can slip into the database. If a model trains on these flags, it will achieve a misleadingly perfect $100\%$ accuracy during testing, but fail completely when deployed live.
To guarantee scientific validity, we built an automated Leakage Shield Engine that executes a Pearson Correlation Sweep across all $51$ candidate numerical features:
$$r = \frac{\sum (X - \bar{X})(Y - \bar{Y})}{\sqrt{\sum (X - \bar{X})^2 \sum (Y - \bar{Y})^2}}$$
Any numerical column that exhibits an absolute correlation coefficient exceeding our threshold ($|r| > 0.98$) with the target column $Y$ is flagged as a target proxy and dropped from the feature space prior to splitting. Constant columns with zero variance are similarly excluded, protecting the feature standardizer from mathematical failures.

3.3 Comparative Technical Analysis
Our XGBoost model won the tournament by securing a peak $F_1$-Score of $0.9868$ ($98.68\%$) and an accuracy of $99.56\%$. Its sequential gradient boosting architecture minimizes error margins by continuously targeting misclassified residual weights from previous tree iterations.
In stark contrast, Gaussian Naive Bayes performed poorly, registering a baseline $F_1$-score of only $48.03\%$ and a low precision of $31.81\%$. The technical explanation for this failure lies in the conditional independence assumption of Naive Bayes:
$$P(X_1, X_2, \dots, X_n \mid Y) = \prod_{i=1}^n P(X_i \mid Y)$$
In real-world network traffic, features such as packet size, packet count, and session duration are highly correlated and mutually dependent. Naive Bayes' mathematical assumption of absolute independence is violated across these network flow metrics, leading to an over-sensitivity to background variations and a high rate of false positives.
4. Technical Trust, Explainability, & Fairness Audit
4.1 Explainable AI (XAI) Baseline
To avoid the risks of a "black-box" model, we mapped out feature importance drivers using SHAP (SHapley Additive exPlanations) values, which are rooted in cooperative game theory. This approach calculates the marginal contribution of each feature across all possible feature combinations.
Our SHAP audit confirmed that Packet Asymmetry was the primary feature driver of anomalous classifications (contributing $45\%$ of overall model output variance), followed by the Destination Port ($30\%$) and Backward Packets per Second ($15\%$).
4.2 Algorithmic Fairness Audit
In network engineering, a primary risk of automated blocking is the potential to block critical background system protocols (e.g., DNS on port 53, or HTTPS on port 443) due to a false positive classification. To audit this risk, we conducted a statistical parity check across two distinct groups of network connections:
Group A (System Ports): All connection records targeting well-known standard ports ($\le 1023$).
Group B (Dynamic Ports): All connection records targeting dynamic, user-defined ports ($> 1023$).
We calculated the False Positive Rate ($FPR$) for both groups to check for bias:
$$FPR = \frac{FP}{FP + TN}$$
Our audit returned an empirical $FPR$ of $0.00\%$ across both Group A and Group B. This statistical parity demonstrates that our model is highly fair and does not exhibit bias toward blocking legacy network services or custom application layers.
5. Deployment, MLOps, & GenAI Integration
5.1 FastAPI Microservice Layer
To transition our trained model from an offline research script to a production service, we wrapped the champion XGBoost binary (intrusion_detector.joblib) and our standardizer (scaler.joblib) into a high-speed FastAPI web framework. This microservice exposes a secure REST API endpoint (/predict) that accepts incoming JSON payloads representing active packet captures:
{
  "features": [31537, 2, 66, 33, 33, 33, 0, 49, 0, 0.95, ...]
}
The endpoint processes the input features and returns an instantaneous classification response in under $25\text{ ms}$, enabling real-time inline evaluation at gigabit speeds.
5.2 Docker Containerization
To guarantee portability across any operating system or cloud architecture, we containerized our microservice using Docker. Our Dockerfile pulls a lightweight Python base image, installs the dependencies listed in our requirements.txt, copies our model binaries, and exposes port 8501. This allows security teams to deploy the entire CyberFlow IDS anywhere in the cloud with a single command:
docker run -p 8501:8501 cyberflow-ids-image
5.3 GenAI SOC Copilot Integration
To eliminate the issue of alert fatigue in Security Operations Centers, we built a GenAI SOC Copilot that uses the Gemini API.
Instead of forcing a human analyst to manually investigate anomalous logs, the system automatically bundles key metadata from any flagged attack—including the engineered Packet_Asymmetry metric, the target destination port, and the active SHAP feature weights—into a structured context prompt. The generative model processes this prompt to return a human-readable Threat Intelligence Brief in real time:
🔴 CRITICAL ALERT: Volumetric Port Scan Detected
• Threat Vector: Automated reconnaissance scan targeting dynamic application layers.
• Key Indicator: Packet Asymmetry is highly skewed (+0.95), suggesting unilateral traffic sweeps.
• Immediate Action: Blacklist source IP 192.168.1.108 and rate-limit ingress handshakes on port 80.
This integration bridges the gap between raw machine learning metrics and practical, human-led incident response, bringing automated security auditing to the modern enterprise.
